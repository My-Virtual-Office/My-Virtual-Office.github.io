Chat service, in detail

Backend

Chat-service owns every kind of conversation in the platform — group channels, DMs, and the chat panel behind a voice room — as one Mongo document model, not three. A channel's type is GROUP, DIRECT, or ROOM; everything downstream (threads, reactions, pins, read receipts) works the same way regardless of which kind you're in.

Three collections, one shape

  • Channels — a DM is keyed by {min-userId}_{max-userId} so the two participants always land on the same document, no matter who started it. A ROOM channel's Mongo _id is set equal to the room's id in room-service — one join, no lookup table. Every workspace gets two channels provisioned automatically: #general and a #join feed for who's arrived.
  • Messages — reactions as a map of emoji to user ids, attachments backed by GridFS, and a captured senderRole so "an admin can't delete another admin's message" still holds even after someone's role changes later.
  • Threads — flat, one level: channel → thread → messages. A thread doesn't nest inside another thread.

Two ways to reply, on purpose

threadId means the message lives in a thread panel. replyToId means it's an inline quote-reply, still in the main channel. They're not the same feature wearing two names — a thread branches the conversation; a reply just points backward in one.

Real-time: one broadcast, not a fan-out guess

Every mutation — a new message, an edit, a reaction, a pin, a delete — goes out over STOMP the moment it happens. A message inside a thread broadcasts only to that thread's topic; it doesn't also land in the channel feed and get filtered client-side. That sounds like a small distinction; it's the difference between a thread reply that never appears where it shouldn't, and one that flashes in the main channel for a frame before disappearing.

The WebSocket handshake can't carry an Authorization header — browsers don't allow it — so the client asks for a one-time Redis-backed ticket first, then presents it on connect. The ticket is consumed atomically on redemption, so two connections racing on the same ticket can't both get in.

A mention is a message, once

The client tells chat-service which user ids were mentioned; chat-service doesn't re-parse the text. One notification event goes out per mention, over RabbitMQ, and notifications-service turns it into whatever the recipient sees. Chat-service never emails anyone directly — it produces one event and moves on.

Chat-service is a consumer too

Create a workspace, and a listener provisions #general and #join without chat-service ever being asked directly — it hears the workspace-created event and acts on it. Create a voice room, and the matching chat channel appears the same way. The 2D office's in-world chat persists through this exact same pipeline: same channels, same history, same rules — not a separate chat system that happens to look similar.
