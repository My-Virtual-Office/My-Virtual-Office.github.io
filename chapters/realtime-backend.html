Rooms & notifications

Backend

Room-service doesn't carry a single byte of audio or video — Agora does that. What it owns is the decision layer around a call: who's in which room, whether the room is full, what their mute state is, and — the part that makes the 2D office feel like a real place — which cluster of nearby avatars should actually hear each other.

Presence lives in Redis, and only Redis

A participant's mute/camera/screen-share state sits in a Redis hash, and a short-lived liveness key with a 30-second TTL, refreshed by a client heartbeat. Nothing about live presence touches a database — if a browser tab dies without saying goodbye, the liveness key simply expires and the next read quietly drops them. Joining a room runs one atomic Lua script, so two people racing for the last seat in a capped room can't both get in.

Proximity voice, computed from positions

The 2D client streams avatar positions; room-service clusters the ones within hearing range using union-find, and gives every cluster of two or more its own private Agora channel. Step out of range and the next snapshot recomputes the group — only the avatars whose group actually changed get told. A named zone (a meeting room, drawn on the floor plan) overrides open-area clustering entirely: step inside one and you join that zone's channel regardless of who's standing next to you.

Notifications: nine event types, one decision per type

Five services publish into a single notification pipeline, and every event carries a unique id so a retried delivery never sends twice. Each type gets exactly one routing decision, made once:

EventDelivery
Signup, OTP, password reset, workspace inviteEmail only
Task assignedIn-app only
Task reminder, mentionIn-app only
Meeting reminderBoth — in-app and email
Said plainly: the routing table above has a spot reserved for a "login succeeded" email that nothing currently sends — the template and the route both exist, but no service publishes that event today. We'd rather say so than let the table imply more than the system does.

Email delivery is deduplicated in Redis against the event's id for 24 hours, and the claim is released if the send actually fails — so a retry gets a real second attempt instead of being silently swallowed by its own idempotency guard.
