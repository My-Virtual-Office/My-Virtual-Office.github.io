Identity & security

Backend

Identity is decided exactly once, at the edge. The gateway is the only thing on the network that gets to say who you are — every service behind it just believes the header it's handed, because nothing else is allowed to set that header.

What the gateway does on every request

  • Strips any client-supplied X-User-Id, X-User-Role, X-User-Email — a client claiming to be someone else gets overwritten, not trusted.
  • Blocks /api/internal/** with a 403 at the edge — the internal API service-to-service calls use is unreachable from outside, full stop.
  • Validates the JWT (HS256, shared secret with user-service) and, on success, injects X-User-Id, X-User-Role and X-User-Email as headers a downstream service can actually trust — because it now knows nothing else could have set them.
  • A short allowlist of paths (login, OTP, the WebSocket handshakes) pass through unauthenticated, still stripped of any spoofed header.

Downstream, every service reads identity through one small helper that pulls X-User-Id off the request — no service parses a JWT, no service owns a session. If the header's missing or malformed, that's a 401 or 400 before any business logic runs.

Two kinds of authorization

The gateway proves who you are. It doesn't know if you're allowed to do the thing you're asking for — that's workspace-scoped, and it's checked by the service that owns the resource. Chat asks workspace-service "is this user a member, and what's their role?" before it lets you create a channel. Calendar does the same before it lets you edit someone else's event. There's no platform-wide admin role baked into the JWT — X-User-Role is always the constant USER; every real permission decision happens per-workspace.

The internal API

Some checks have to be synchronous — a chat message can't wait for an event queue to confirm the sender is still a member. Those calls go over a locked-down internal API, guarded by a shared X-Internal-Token header and blocked at the gateway so it's server-to-server only. Callers fail closed: if workspace-service is unreachable, the answer is 503, never an implicit yes.

Where we've drawn the line, honestly

A symmetric JWT secret is shared across the gateway, user-service, and the 2D office's Colyseus server — whoever holds it can mint a token. That's a real trade-off, not an oversight: moving to asymmetric signing (RS256 + JWKS) is the obvious next step, and we're saying so here instead of pretending the current setup is the final answer.

Every credential this system needs — the JWT secret, database passwords, the Agora certificate, the SMTP password — is read from the environment, not committed to the repository. The deployment script that packages a release generates fresh infrastructure secrets per machine; nothing ships with a real credential baked in.
